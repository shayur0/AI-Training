# Documentation conventions — Illumination Space

This repo has four living documents, each with a specific format that exists
for a specific reason (see `notes/memory.md` for the lesson behind each one).
Don't freelance the structure even for a "small" edit.

## instructions/planning.md — the 9-stage pipeline

Every stage entry MUST use exactly this structure — do not collapse it into
prose:

```
## Stage N: <Name>

**Inputs:** ...
**Actions:** ...
**Outputs:** ...
**Considerations/Risks:** ...
```

Why: forcing every stage through this structure is what originally surfaced
risks that prose would have glossed over (e.g. the live-data fallback ladder
in Stage 2 only got written down because "Risks" was a mandatory section, not
an afterthought). Preserve this even when a stage seems simple enough to
describe in a paragraph.

Section 6 ("Open Decisions for the Platform Owner") tracks decisions that are
still open vs. resolved — when a decision gets resolved, strike it
(`~~text~~`) and add a `**RESOLVED (...)**:` line inline, AND add a
corresponding entry to `decisions/decision.md` (see below). Keep both in
sync; don't resolve in one file without the other.

## decisions/decision.md — the decision log

Each entry is exactly three fields, one sentence each:

```
## N. <Short title>

**Decided:** <one sentence — what we're doing>

**Why:** <one sentence — the load-bearing reason>

**Ruled out:** <one sentence — the alternative(s) rejected and why>

(Source: [planning.md](../instructions/planning.md) <Stage/Section reference>)
```

Why one sentence per field: the log is for fast lookup later, not for
re-deriving the reasoning from scratch. When drafting a new entry, write the
full reasoning first if it helps, then cut each field down to the single
clause that's actually load-bearing — don't just truncate the first sentence
of a paragraph.

Always link back to the source stage/section in planning.md.

## notes/memory.md — the session journal

Each entry:

```
## YYYY-MM-DD — <short task title>

**Task:** <what was done this session>

**Lesson:** <one true lesson learned — not a summary of the task>
```

Why: this is a lesson log, not a changelog. "Lesson" should be something that
would change future behavior if read cold — a surprising failure mode, a
format choice that paid off, a thing that silently went stale. If the
"Lesson" field just restates the "Task" field, it's not done yet. Add one
entry per session that changes this repo's content; don't batch multiple
sessions into one entry, and don't skip writing one because the session felt
minor — the 2026-07-06 entry about `structure.md` going stale is exactly the
kind of lesson that's only useful because it got written down.

## structure.md — the repo map

Not a generated file; it drifts silently. Before trusting it, or right after
adding/removing/renaming a top-level folder, diff it against reality:

```bash
find . -maxdepth 1 -type d -not -path '*/.git*' | sort
```

Update the "Content folders" or "Tooling / agent config (dotfiles)" section
to match. See the `structure-sync` agent for an automated version of this
check.

## CLAUDE.md / AGENTS.md

These two files are independent (not symlinked, no shared inode) but must
stay substantively in sync — the Beads-integration block in particular is
managed/regenerated by `bd setup`. If you hand-edit one, mirror the
non-generated parts to the other.
